# 项目制学习系统 · 完整蓝图与落地计划

> 基于 2026-04-28 产品讨论，为「奥数练习册 AI 错题集」新增「项目制学习」系统

---

## Context

**当前问题**：错题集是单一工具，学生随机上传，无目标驱动，无法形成学习闭环，用户黏性差。

**解决方向**：在现有错题集基础上，叠加「项目制学习」框架，让每一道错题都归属于一个有明确目标、有截止日期的学习项目中，使错题从「随机记录」变为「进度反馈」。

**通用框架（长期愿景）**：
```
定位 → 目标 → 计划 → 执行 → 复盘
```
首个项目为「小学奥数华杯备赛」，成熟后平滑扩展到其他学习目标（雅思 100 天、期末冲刺等）。

---

## 产品哲学与意义体系

**方法论是产品价值内核，华杯只是第一个载体。**

学生通过「项目制学习」真正习得的，是一套面对复杂任务的心智模型：
> 把一项复杂的、甚至看似难以完成的目标，拆解成一个个可执行的任务单元，逐步推进，直至最终达成——而往往在这个过程中，他们会发现：结果本身没有那么重要了，重要的是他们已经拥有了应对下一个挑战的能力。

**产品演进路径：**

- **Phase 1（当前）**：系统「推着走」——学生在 AI 引导下走完完整的项目周期，通过重复建立方法论认知
- **Phase 2（远期）**：用户自发布学习项目，形成生态飞轮，运营成本下降，项目资源持续丰富

**家长购买逻辑：**

家长买单的直接理由是备赛结果，但实质是对方法论的信任：「相信这套方法能更好地保障孩子的成绩」。因此产品对外的语言体系应强调「思维能力培养」而非「学科竞赛备考」，既符合双减语境，也更准确地传达产品的长期价值。

### 用户完整旅程

```
1. 定位阶段
   └── AI 前置沟通（年级、奥数经历等）→ 初步能力判断

2. 目标阶段
   ├── 选择项目类型（华杯备赛）→ 设定截止日期 → 选择目标级别
   └── 驱动目标锚点确立：「我要在 X 月 X 日拿到 XX 分」
       持续显示于首页顶部 / 每日任务面板顶部（全程可见，心理锚点）

3. 评估阶段
   ├── AI 分配诊断套题（按初步能力匹配）
   ├── AI 自动判题
   ├── 生成「能力报告」（整体层次 + 各知识点薄弱程度）
   ├── AI 与学生讨论报告 → 能力校正 → 确定最终能力画像
   └── 【战略地位】能力报告是产品「第一信任节点」（零时刻）
       报告须同时传达两层信息：
       ① 「了解你的孩子」—— 个性化数据，非通用标签
          示例：「你在余数类题目正确率为 62%，低于同级均值 23%，这是重点突破区」
       ② 「科学有依据」—— 呈现方法论的系统性与专业感，建立家长对结果的理性预期

4. 计划阶段
   ├── AI 基于「能力画像 × 知识体系 × 剩余天数」生成个性化计划
   ├── 计划包含：阶段里程碑 + 每日知识点任务
   └── 学生/家长确认计划（或微调）→ 计划生效

5. 执行阶段（衔接现有错题集）
   ├── 日历可视化：每日任务清晰呈现
   ├── 学生学习 → 上传作业/错题（现有流程）
   ├── 日历颜色追踪完成状态（已有）
   ├── 遗留任务可见，学生自行补救
   ├── 微信每日学习报告推送给家长
   └── 里程碑完成 → 系统自动触发「里程碑复盘」（无需人为触发）
       ├── AI 提问 3 题：最难的题 / 掌握感受 / 下阶段期待
       ├── 生成「阶段成就卡」+ 解锁里程碑勋章（打怪通关式即时反馈）
       └── 成就卡归档至项目档案，供最终复盘回看完整成长轨迹

6. 复盘阶段
   ├── 学生个人复盘：学习轨迹 + 成长总结 + 下次建议
   └── 平台运营分析：跨学生横向对比，优化计划模板
```

## 新增数据模型

### 新增数据库表（在现有 schema 基础上扩展）

```prisma
// 项目类型（平台运营维护）
model ProjectType {
  id          String   @id
  name        String   // "华杯小学奥数", "雅思备考"
  subject     String   // "math", "english"
  gradeRange  String   // "4-6", "all"
  description String
  isActive    Boolean  @default(true)
  
  knowledgeSystems KnowledgeSystem[]
  projects         LearningProject[]
}

// 知识体系（AI 初始化 + 运营充实）
model KnowledgeSystem {
  id            String      @id
  projectTypeId String
  version       String      // "v1.0"
  
  projectType   ProjectType @relation(...)
  nodes         KnowledgeNode[]
}

// 知识点节点
model KnowledgeNode {
  id               String          @id
  systemId         String
  name             String          // "余数基本运算"
  parentId         String?         // 支持层级结构：领域 → 章节 → 知识点
  level            Int             // 1=领域, 2=章节, 3=知识点
  difficulty       Int             // 1-5
  prerequisites    String[]        // 前置知识点 IDs
  estimatedDays    Int             // 建议学习天数
  
  system           KnowledgeSystem @relation(...)
  parent           KnowledgeNode?  @relation(...)
  children         KnowledgeNode[] @relation(...)
}

// 学习项目实例（每个学生的具体项目）
model LearningProject {
  id            String        @id
  studentId     String
  projectTypeId String
  targetDate    DateTime      // 考试/目标截止日
  goalLevel     String?       // "初级/中级/高级" 目标
  status        ProjectStatus // intake | assessing | planning | confirmed | executing | retrospecting | completed
  
  student       Student       @relation(...)
  projectType   ProjectType   @relation(...)
  assessment    AbilityAssessment?
  studyPlan     StudyPlan?
  retrospective ProjectRetrospective?
  createdAt     DateTime      @default(now())
}

enum ProjectStatus {
  intake        // AI 前置沟通中
  assessing     // 诊断测试中
  planning      // AI 生成计划中
  confirmed     // 计划已确认，待执行
  executing     // 执行中
  retrospecting // 复盘中
  completed     // 已完成
}

// 能力评估
model AbilityAssessment {
  id          String         @id
  projectId   String         @unique
  intakeData  Json           // AI 前置沟通收集的信息
  abilityLevel String        // "beginner" | "intermediate" | "advanced"
  weakPoints  Json           // { nodeId: String, score: Int }[]
  reportData  Json           // 完整能力报告内容
  calibrated  Boolean        @default(false)
  
  project     LearningProject @relation(...)
  sessions    AssessmentSession[] // 诊断题答题记录
}

// 诊断题答题会话
model AssessmentSession {
  id           String            @id
  assessmentId String
  questions    Json              // 题目列表 (来自现有 Chapter 体系)
  answers      Json              // 学生答案
  aiGrading    Json              // AI 判题结果
  completedAt  DateTime?
  
  assessment   AbilityAssessment @relation(...)
}

// 学习计划（总计划）
model StudyPlan {
  id           String          @id
  projectId    String          @unique
  generatedAt  DateTime
  confirmedAt  DateTime?       // 学生/家长确认时间
  totalDays    Int
  planData     Json            // 完整计划数据快照
  
  project      LearningProject @relation(...)
  milestones   Milestone[]
  dailyTasks   DailyTask[]     // 与现有 TrainingPlan 对齐
}

// 里程碑
model Milestone {
  id          String     @id
  planId      String
  name        String     // "阶段一：整除与余数"
  startDate   DateTime
  endDate     DateTime
  nodeIds     String[]   // 覆盖的知识点 IDs
  status      MilestoneStatus
  
  plan          StudyPlan               @relation(...)
  retrospective MilestoneRetrospective?
}

enum MilestoneStatus {
  pending
  in_progress
  completed
  overdue
}

// 里程碑复盘（系统自动触发）
model MilestoneRetrospective {
  id           String    @id
  milestoneId  String    @unique
  triggeredAt  DateTime  // 系统自动触发时间
  qaData       Json      // AI 3问 + 学生回答
  badgeAwarded String    // 获得的勋章标识
  cardData     Json      // 阶段成就卡内容快照

  milestone    Milestone @relation(...)
}

// 每日任务（扩展现有 TrainingPlan）
// 保持 TrainingPlan 现有结构，新增字段关联 StudyPlan
// 在现有 TrainingPlan 上追加：
//   planId      String?   // 关联 StudyPlan
//   nodeIds     String[]  // 对应知识点
//   milestoneId String?   // 所属里程碑

// 项目复盘
model ProjectRetrospective {
  id              String          @id
  projectId       String          @unique
  studentReport   Json            // 学生个人复盘数据
  platformMetrics Json            // 平台运营数据
  generatedAt     DateTime
  
  project         LearningProject @relation(...)
}
```

## 分阶段落地计划

### Phase 1：知识体系 + 项目创建（预计 2 周）

**目标**：学生能选择「华杯备赛」项目，设定截止日期，完成 AI 前置沟通

**交付物**：
- [ ] DB 迁移：ProjectType, KnowledgeSystem, KnowledgeNode, LearningProject 表
- [ ] 导入「华杯小学奥数」知识体系（AI 生成初版 ~50个知识点节点，运营审核）
- [ ] 后端：`/projects` CRUD + `/knowledge-systems` API
- [ ] 前端：`project/setup` 页（首次使用引导，设定华杯截止日期，直接进入 intake）
- [ ] 前端：`project/intake` 页面（结构化问卷，4-6 个固定问题）
- [ ] home 页：未开始时显示「开始备赛」引导入口；进行中时 home 直接是执行视图

**核心数据**：华杯奥数知识体系 Layer 1（按「领域 → 章节 → 知识点」三层结构）

---

### Phase 2：能力评估流程（预计 3 周）

**目标**：学生能完成诊断测试，获得「能力报告」，并与 AI 讨论校正

**交付物**：
- [ ] DB 迁移：AbilityAssessment, AssessmentSession 表
- [ ] 诊断题套题设计（华杯小学奥数 3 套，低/中/高能力级各一套，每套约 15–20 题，AI 前置沟通后分配对应套题，复用现有 Chapter/Problem 结构）
- [ ] 后端：`/projects/:id/assessment` 接口（提交答题 → AI 判题 → 生成报告）
- [ ] 后端：`/projects/:id/calibrate` 接口（AI 对话校正能力画像）
- [ ] 前端：`project/assessment` 做题界面（复用现有 problem-detail 交互组件）
- [ ] 前端：`project/report` 能力报告页（层次 + 薄弱点可视化 + AI 对话框）
      **报告设计原则**：个性化数据（非通用标签）+ 方法论专业性 = 家长「第一信任节点」
- [ ] AI Prompt：能力报告生成（结构化 JSON 输出）
- [ ] AI Prompt：校正对话（Socratic 追问风格）

---

### Phase 3：个性化计划生成（预计 2 周）

**目标**：AI 基于能力画像生成个性化日历计划，学生/家长确认后生效

**交付物**：
- [ ] DB 迁移：StudyPlan, Milestone, TrainingPlan 字段扩展
- [ ] 计划生成算法（规则引擎 v1：能力层次 → 计划模板 → 微调薄弱点天数）
- [ ] 后端：`/projects/:id/generate-plan` 接口
- [ ] 后端：`/projects/:id/confirm-plan` 接口（批量写入 TrainingPlan）
- [ ] 前端：`project/plan-preview` 页面（里程碑视图 + 每日任务预览）
- [ ] 现有 `home` 日历页：展示项目计划来源标记（区分系统生成 vs 手动）
- [ ] 计划模板设计（华杯3个层次 × 4个时间周期 = 12个基础模板）

**算法核心**：
```
输入：知识节点列表（带权重薄弱程度）+ 剩余天数 + 能力层次
输出：按日期排列的知识点学习序列

规则：
1. 按知识点前置依赖关系拓扑排序
2. 薄弱点 → 分配更多天数
3. 每日任务量上限：2-3 个知识点
4. 预留缓冲天数（约 20%）用于复习和追赶遗留任务
```

---

### Phase 5：复盘系统（预计 2 周）

**目标**：项目结束后自动生成复盘报告，支持学生成长沉淀和平台数据分析

**交付物**：
- [ ] DB 迁移：ProjectRetrospective 表
- [ ] 后端：复盘报告生成逻辑（聚合执行数据 + AI 生成文字总结）
- [ ] 前端：`project/retrospective` 复盘报告页
- [ ] 平台侧数据看板（运营内部工具，非小程序）：跨项目完成率、薄弱点分布

**复盘报告内容**（学生侧）：
```
- 项目总览：目标 / 完成时间 / 整体完成率
- 知识掌握变化：开始前 vs 结束后薄弱点对比
- 执行亮点：连续完成最长记录、最快攻克的知识点
- 成长建议：AI 生成下次备赛优化方向（3条）
```

## 关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 计划是否自动调整 | 不自动调整，计划固定 | 培养学生自律；遗留可见即为压力 |
| 诊断题来源 | 运营手工整理（复用现有题目结构），不用历年真题 | 避免版权风险；可控质量 |
| AI 对话交互方式 | 小程序内结构化问卷（本期），企微机器人（未来） | 降低开发复杂度，先验证流程 |
| 计划生成方式 | 规则引擎 v1（模板 + 薄弱点微调），非纯 AI | AI 成本可控；可解释性强；后续数据积累后升级 |
| 家长视角 | 与学生共用账号/视角 | 小学生无手机，家长手机即学生账号 |
| 里程碑复盘触发方式 | 系统自动触发，无需人为 | 降低使用门槛，保证每次都发生 |
| 里程碑复盘体验形态 | 游戏化（AI 3问 + 勋章 + 成就卡）而非普通任务 | 仪式感 + 即时反馈提升成就感，避免流程化敷衍 |
| 本期不做 | 计划变更、企微机器人、多学科 | 记录为后续优化方向 |

## 后期迭代核心方向

> 以下内容在蓝图讨论阶段有意搁置，颗粒度超出当前阶段范畴，但均有展开讨论的价值，待 Phase 1–5 稳定后逐项提上议程。

### 1. 游戏化激励机制深化

当前蓝图确定了「成就勋章 + 升级解锁」的游戏化方向，但以下细节待定：

- **解锁的是什么**：纯视觉奖励（头像框、勋章展示）vs 实质性内容解锁（更高难度题目、新知识模块、特殊 AI 教练风格）
- **小学生激励心理研究**：不同年龄段（8–12岁）对哪类即时反馈最敏感
- **勋章体系设计**：勋章命名、视觉风格、稀有度分层
- **跨项目成就**：完成多个项目后的累计成就体系

### 2. 评价体系的独立定义

讨论中提出：评价体系可能需要从知识体系中独立出来，单独定义「什么是好的任务完成」「达成最终目标需要好到什么程度」。

待探讨的核心问题：

- 评价体系与知识体系、能力画像的关系边界如何划分？
- 每个知识点的「掌握标准」（Mastery Level）如何量化？
- 里程碑目标的通过门槛如何设定？
- 最终目标（华杯分数）与各阶段评价指标之间的映射关系

### 3. Phase 2 用户发布生态

当前 Phase 1 以平台主导的「推着走」模式为主。Phase 2 用户自发布学习项目的生态设计待讨论：

- **发布者身份**：学生 / 家长 / 老师 / 教育机构，各自的发布动机与质量保证机制
- **项目质量审核**：UGC 内容的审核标准与流程
- **发布激励机制**：发布者获得什么回报（流量分成、声誉积分、订阅分成）
- **生态冷启动**：从平台主导到 UGC 的过渡策略

### 4. 方法论内化路径设计

当前阶段目标是让学生「被推着走完项目」。如何从「推着走」过渡到「主动内化方法论」，需要专项设计：

- 什么信号表明学生已内化方法论（可以自主拆解新任务）？
- 是否需要设计「渐进放权」机制：前几个项目全程引导，后续项目逐步减少 AI 介入？
- 方法论可视化：是否在某个时机向学生和家长显式呈现「你已经掌握了这套方法」？
