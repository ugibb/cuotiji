## 第 17 轮

🤖 **AI**

好，我们现在进入 OCR 具体实现的讨论。先把技术现状摆清楚，再逐步推演方案。

**关键发现：现有架构已经很好**

当前 UI 设计是**一题一拍**（每个 slot 独立拍照），这对 OCR 是最有利的结构——每张图只有一道题的作答，AI 不需要分割页面。

**但现在有一个重要的缺口：**

`daily/index.ts:161` 提交时只传了 `chapterId, planDate, imageUrl, studentId`，**没有传 `planItemId` 和 `questionText`**。服务端收到图片时不知道这是哪道题，OCR 只能靠图片猜题意。

**实现方案（四步）**

### Step 1：拍照对象讨论

**方案 A：只拍答案区域**
- 学生在稿纸/题目旁边写答案，拍解题过程那块
- OCR 只识别手写内容，题目文字已知

**方案 B：拍题目 + 答案整体**
- 连题目带答案一起拍
- OCR 可验证拍的是不是对应这道题（mismatch 检测）
- 但图片信息更杂，OCR 干扰更多

**我的建议**：方案 B，保留 mismatch 检测逻辑（你们已经写了，不要浪费），但 OCR 时用已知题目文字辅助 AI 过滤噪音。

---

### Step 2：OCR API 选型

| API | 适用场景 | 奥数答题 |
|-----|---------|---------|
| `GeneralHandwritingOCR` 通用手写体 | 混合中文+数字 | **推荐** |
| `GeneralAccurateOCR` 通用精准 | 印刷体 | 不适用 |
| Mathpix | 数学公式LaTeX | 过度设计 |

---

### Step 3：服务端改造（2处）

**① `ocr.service.ts`** — 替换 mock

```typescript
// 新签名：接收 imageUrl，返回原始识别文字
async function extractHandwriting(imageUrl: string): Promise<string>
```

**② `processAssignment` 逻辑改造**

```
新流程：
  传入 knownQuestion（已知题目）
  OCR → 提取所有手写文字（raw text）
  AI  → 接收 {knownQuestion + rawOcrText} → 解析出学生答案 + 批改
```
